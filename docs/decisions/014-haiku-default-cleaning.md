# ADR-014: Haiku default для очистки транскриптов

## Статус

Принято (v0.65), обновлено (v0.66, v0.72, v0.73)

## Контекст

Очистка транскриптов — механическая задача: удаление слов-паразитов, исправление терминологии по глоссарию. До v0.65 для очистки использовался Claude Sonnet — та же модель, что и для аналитических задач (longread, summary).

Тестирование показало, что Haiku и Sonnet дают сопоставимый результат на задаче очистки:
- Идентичное качество удаления паразитов
- Идентичное исправление терминологии по глоссарию

При этом Sonnet стоит $3/$15 за 1M токенов, а Haiku — $1/$5 (3x дешевле).

## Решение

Изменить default модель для очистки с `claude-sonnet-4-6` на `claude-haiku-4-5`.

Пользователь может переопределить через:
- ENV: `CLEANER_MODEL=claude-sonnet-4-6`
- Web UI: выбор модели в пошаговом режиме

## Обновление v0.66: Упрощение промпта до 2 задач

### Контекст

В v0.65 промпт очистки содержал 3 задачи:
1. Удаление слов-паразитов
2. Исправление терминологии по глоссарию
3. Причёсывание текста (удаление дублей, заиканий, техшума)

Тестирование на полном транскрипте (76K символов, ФСТ Мужской клуб) выявило:
- **Задача 3 игнорируется** Haiku на длинном тексте (40K-чанки). Дубли, заикания, техшум остаются в выходе
- Даже Sonnet не справляется с удалением дублей на 40K-чанках — модели хорошо делают точечные замены, но не могут "выкинуть" целые фрагменты

### Эксперимент: двухпроходная очистка

Протестирована гипотеза: разделить задачи на два прохода (1-й — терминология с глоссарием, 2-й — очистка без глоссария).

**Тест на фрагменте (3.5K):** оба подхода (один проход с 3 задачами и два прохода) дали сопоставимые результаты. На коротком тексте любая модель справляется хорошо.

**Тест на полном транскрипте (76K):**
- Проход 1 (2 задачи) vs исходный промпт (3 задачи): **идентичные результаты** по терминологии. Убирание задачи 3 не улучшило работу с глоссарием — проблема не в "перегрузке задачами", а в деградации внимания модели на длинных чанках
- Проход 2 (причёсывание): успешно убрал ~8 дублированных фрагментов, подтверждена польза

### Решение

**Оставить один проход с 2 задачами** (паразиты + терминология). Причёсывание (задача 3) убрано из промпта.

Причины:
- Задача 3 всё равно не выполнялась на длинном тексте
- Оставшиеся дубли и шум **не критичны** — longread/story LLM перерабатывает текст и передаёт смысл корректно даже с артефактами устной речи
- Двухпроходная очистка (x2 стоимости) не оправдана: основной шум — ошибки Whisper-транскрибации (бессмысленные фразы), которые LLM не может исправить ни за один, ни за два прохода
- Стратегическое направление: улучшение качества на уровне транскрибации (модели Whisper, качество аудио), а не пост-обработки

### Нерешённые проблемы (ограничения Whisper)

На полном транскрипте остаются ошибки, которые LLM-очистка не может исправить:
- Бессмысленные фразы: "военно-водочной силы", "Низкий Покром нашла решение", "на диагнозе десятки этих денег"
- Искажения без контекста: "Роналду и Сталин", "способские отношения" (→ спонсорские)
- Термины, отсутствующие в глоссарии: "Дебайв" (→ Herbalife?)

Это ограничение Whisper, не очистки. Путь улучшения — лучшие модели транскрибации или предобработка аудио.

## Последствия

### Положительные
- ~3x снижение стоимости очистки (основной потребитель токенов в pipeline)
- Более быстрая обработка (Haiku быстрее Sonnet)
- Промпт проще и фокуснее — модель не тратит контекст на невыполнимую задачу

### Отрицательные
- Дубли и заикания остаются в очищенном транскрипте (не критично для longread/story)

### Нейтральные
- Аналитические задачи (longread, summary) остаются на Sonnet
- Все модели Claude остаются доступны для выбора в UI

## Обновление v0.72: Фокус на глоссарной подстановке

### Контекст

Тестирование на транскрипте 60K символов (ФСТ Дмитрук Светлана) выявило баг: текст вырос на +3.3% вместо уменьшения. Причина — overlap-зона (3000 символов) обрабатывалась моделью по-разному в двух чанках (разная капитализация «Независимым»/«независимым»), overlap-мерж не находил точное совпадение, весь overlap попадал в результат дважды.

Дополнительно: удаление слов-паразитов на длинных текстах работает слабо (Haiku пропускает большинство), а longread-модель (Sonnet) устраняет паразиты сама при генерации.

### Решение

**Оставить только 1 задачу: глоссарная подстановка терминов.** Убрать задачу удаления слов-паразитов и overlap между чанками.

Изменения:
- Промпт `cleaning/system.md` — 1 задача (терминология) вместо 2 (паразиты + терминология)
- `TranscriptCleaner._split_into_chunks()` — без overlap
- `TranscriptCleaner._merge_chunks()` — простая конкатенация
- `num_predict` увеличен (output ≈ input)
- Пороги валидации ужесточены (>15% = warning, >25% = error)
- `chunk_overlap: 0` во всех context_profiles

### Последствия

- **Устранён баг с дублированием** — overlap-зоны больше нет, мерж тривиален
- **Более надёжная очистка** — модель фокусируется на одной точечной задаче
- **Слова-паразиты остаются** в очищенном транскрипте, но longread/story LLM их устраняет
- **Output ≈ input по длине** — разница только в заменённых терминах (±2-3%)

## Обновление v0.73: Canonical с аббревиатурами для RAG-поиска

### Контекст

Тестирование diff-подсветки (v0.73) выявило две проблемы:
1. **Модель подставляла аббревиатуры без контекста**: `canonical: "ЭО"` → в очищенном тексте "ЭО" без расшифровки. Longread-модель не получает глоссарий и не знает что "ЭО" = "Экспресс-оценка". Аббревиатура "ПО" вообще двусмысленна (Полная оценка vs Программное обеспечение).
2. **RAG-поиск**: если canonical содержит только полное название ("Экспресс-оценка"), чанки лонгрида не будут содержать аббревиатуру "ЭО" — поиск по сокращению не работает.

### Решение

**Canonical в формате "Полное название (аббревиатура)"** для терминов с часто используемыми сокращениями.

Примеры:
- `canonical: "Экспресс-оценка (ЭО)"` — вместо просто "ЭО"
- `canonical: "Супервайзер (SV)"` — с английской аббревиатурой из BizWorks
- `canonical: "Встреча Возможностей для Клиентов (ВВК)"` — раскрытие + аббревиатура

Промпт дополнен правилом точной подстановки: "если canonical содержит скобки — подставляй целиком, допускается склонение основного слова".

### Цепочка данных

```
Whisper → "экспресс-оценку" → Очистка → "Экспресс-оценку (ЭО)" → Longread → понимает контекст, сокращает при повторах → Chunks → RAG-поиск по "ЭО" и "Экспресс-оценка"
```

### Последствия

- **Longread-модель получает контекст** — видит "Экспресс-оценка (ЭО)" и понимает значение без глоссария
- **RAG-поиск работает** по обоим вариантам (полное название и аббревиатура) в чанках
- **Очищенный текст чуть длиннее** — каждое упоминание содержит скобки, но это промежуточный этап
- **Longread-модель оптимизирует** — при полной переработке текста использует полную форму при первом упоминании, далее сокращает

### Термины с новой canonical формой

Статусы: Супервайзер (SV), Активный Супервайзер (ASV), Ворлд-тим (WT), ГЕТ (GET), Миллионер (MT), Президент (PT).
Объёмы: Групповой объем (GV), Организационный объем (OV).
Роли: Привилегированный клиент (ПК), Независимый партнер (НП).
Мероприятия: Встреча Возможностей для Клиентов (ВВК), Группа поддержки (ГП), Группа роста (ГР), Школа будущих миллионеров (ШБМ), Школа будущих президентов (ШБП).
Сервисы: Экспресс-оценка (ЭО), Полная оценка (ПО).
Продукты: Protein Drink Mix (ПДМ), Овсяно-яблочный напиток (ОЯН), Комплекс пищевых волокон (КПВ).

VP оставлен без изменений — единица измерения, всегда с числом ("2500 VP"), контекст ясен.
