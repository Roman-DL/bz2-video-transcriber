# ADR-015: Single-pass генерация лонгрида

**Статус:** Принято
**Дата:** 2026-02-20

## Контекст

Лонгрид генерировался через map-reduce pipeline, спроектированный под локальные модели (gemma2:9b, 8K контекст): TextSplitter → OutlineExtractor → параллельные секции → combine = 6-12 LLM вызовов.

С v0.29 система использует только Claude (200K контекст), но pipeline всё ещё работал через map-reduce. Типичный текст 43K символов — это 7% контекста Claude. Map-reduce при этом:

1. **Терял контекст** — каждая секция видела только свои части + outline
2. **Создавал точки отказа** — баг `PartOutline.summary < 10` символов (v0.66 hotfix)
3. **Стоил дороже** — overhead на system prompt в каждом из 12 вызовов
4. **Снижал качество** — секции генерировались независимо, повторы неизбежны

## Решение

Авто-выбор стратегии генерации в `LongreadGenerator` по размеру контекста модели:

- **Single-pass** (1 LLM вызов) — когда текст помещается в контекст (Claude 200K)
- **Map-reduce** (существующий код) — когда текст НЕ помещается (Ollama 8-32K)

Выбор автоматический через `_fits_in_context()`:
- Оценка: Russian ~2.5 tokens/char
- Overhead: 15K tokens (промпт) + 30K tokens (выход)
- Утилизация: 85% контекста (safety margin)

Промпты (`system.md`, `instructions.md`, `template.md`) — общие для обоих путей.

## Последствия

### Положительные
- 1 LLM вызов вместо 6-12 для типичного текста
- Модель видит весь контекст — лучше связи между разделами
- Убраны точки отказа outline extraction
- Дешевле (один вызов vs суммарный overhead)
- Универсальность — работает с любыми моделями через авто-выбор

### Отрицательные
- Для текстов >350K символов (3+ часов видео) по-прежнему map-reduce
- Map-reduce код сохранён (поддержка локальных моделей)

## Альтернативы

### 1. Полное удаление map-reduce
- ✗ Потеряли бы поддержку локальных моделей с малым контекстом
- ✓ Проще код

### 2. Принудительный single-pass без проверки контекста
- ✗ Ошибки при использовании с локальными моделями
- ✓ Проще логика выбора
