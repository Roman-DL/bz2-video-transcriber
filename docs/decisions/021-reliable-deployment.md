# ADR-021: Надёжный деплой

**Статус:** Принято
**Дата:** 2026-02-22

## Контекст

Деплой регулярно ломался из-за трёх проблем:

1. **`--no-cache` при каждом build** — Docker пересобирал образы с нуля, заново скачивая `python:3.12-slim`, `node:20-alpine` из Docker Hub
2. **Docker Hub недоступен** — TLS handshake timeout при pull базовых образов (сервер за прокси, нестабильная сеть)
3. **Нет обработки ошибок** — при падении build непонятно что произошло, нет health check после деплоя

Текущие Dockerfile уже оптимизированы для layer cache: `requirements.txt` / `package.json` копируются первыми, код — последними. `--no-cache` обнулял это преимущество.

## Решение

1. **Убрать `--no-cache`** — Docker layer cache корректно инвалидируется при изменении исходников (rsync обновляет файлы → COPY инвалидирует слой)
2. **Pre-pull базовых образов с retry** — перед build проверяем наличие образов; если нет — тянем с 3 попытками (10s пауза между попытками)
3. **Флаг `--pull`** — опциональное принудительное обновление базовых образов (`deploy.sh --pull`)
4. **Health check после деплоя** — 5 попыток проверки `https://transcriber.home/health` с паузой 5s
5. **Вывод build-лога при ошибке** — последние 30 строк для диагностики
6. **Helper-функции `remote()` / `remote_sudo()`** — убрано дублирование SSH-команд
7. **`frontend/.dockerignore`** — исключает node_modules, dist, .git, *.md из build context

## Последствия

### Положительные
- Деплой не падает из-за временных сетевых проблем (retry на pull)
- Build использует layer cache → быстрее в 5-10x (секунды вместо минут)
- При ошибке build — видно последние строки лога
- Health check автоматически проверяет что приложение запустилось
- `--pull` позволяет обновить базовые образы когда нужно (security patches)
- Меньше трафика к Docker Hub при обычном деплое

### Отрицательные
- При обновлении базовых образов нужно явно указать `--pull` (не автоматически)

## Альтернативы

### 1. Docker BuildKit cache mount
- ✗ Требует настройки BuildKit на сервере
- ✗ Избыточно — стандартный layer cache достаточен
- ✓ Ещё быстрее для pip/npm install

### 2. Pre-built images в registry
- ✗ Нужен приватный registry или Docker Hub аккаунт
- ✗ Усложняет инфраструктуру для домашнего проекта
- ✓ Деплой без build вообще

### 3. Оставить --no-cache + увеличить timeout
- ✗ Не решает корневую проблему — каждый деплой тянет ~500MB
- ✗ Медленный деплой даже при стабильной сети
- ✓ Гарантирует чистую сборку
