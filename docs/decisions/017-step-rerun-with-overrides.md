# ADR-017: Перезапуск шагов с настройками

**Статус:** Принято
**Дата:** 2026-02-21

## Контекст

В пошаговом режиме (Step-by-Step) пользователь выполняет pipeline поэтапно и может захотеть перезапустить шаг — например, сменить модель для лонгрида или попробовать другой промпт для очистки. Необходим механизм перезапуска с возможностью выбора модели и варианта промпта.

Система также предусматривает архивный режим (Cache API), где перезапуск работает через версионированный кэш.

## Решение

Два механизма перезапуска для разных сценариев:

### 1. Step-by-Step UI (основной)

Используется в пошаговом режиме обработки:

1. `resetDataFromStep(step)` — сбрасывает данные шага и всех последующих, возвращает `currentStep` к перезапускаемому шагу
2. Пользователь нажимает CTA "Выполнить"
3. `runStep(step)` вызывает `/api/step/{stage}` с `modelOverrides` и `promptOverrides`

Настройки доступны для всех завершённых LLM-шагов (`clean`, `longread`, `summarize`, `story`).

### Разделение UI-навигации и pipeline-состояния

В левой панели два независимых состояния:

- **`activeTab`** — какой результат отображается в правой панели. Клик по завершённому шагу переключает таб. Чисто визуальное состояние.
- **`currentStep`** — на каком шаге стоит pipeline. Определяет CTA ("Выполнить"). Меняется только при завершении шага или нажатии "Перезапустить".

Клик по шагу в левой панели **не** меняет `currentStep` — только переключает просмотр результата. Только кнопка "Перезапустить" двигает pipeline назад.

### 2. Cache API (для архивного режима)

`POST /api/cache/rerun` — перезапуск через версионированный кэш (`.cache/{stage}/v{N}.json`). Предусмотрен для будущего архивного режима, сохраняет версию результата.

### Выбор модели

- `ModelSelector` в панели настроек → `modelOverrides[step]`
- `getModelForStage(stage)` — приоритет: override → settings → default
- Передаётся в API как параметр `model`

### Выбор промпта

- `ComponentPromptSelector` в панели настроек → `promptOverrides[step]`
- `getPromptOverridesForApi(stage)` — формирует объект `{component: variant}` для API
- Селектор виден когда `variants.length > 1`

Варианты промптов сканируются API из двух директорий (приоритет: внешние → встроенные, см. ADR-008):
1. `PROMPTS_DIR/{stage}/` — внешние (на сервере: `/mnt/main/work/bz2/video/prompts/`)
2. `config/prompts/{stage}/` — встроенные в репозиторий

**Как добавить вариант промпта:** положить файл `{component}_v2.md` в `/mnt/main/work/bz2/video/prompts/{stage}/` — UI автоматически покажет селектор.

## Последствия

### Положительные
- Пользователь может экспериментировать с моделями и промптами без перезапуска всего pipeline
- Настройки доступны для любого завершённого LLM-шага, не только текущего
- Добавление варианта промпта — просто файл на диске, без изменения кода

### Отрицательные
- Два механизма перезапуска — потенциальная путаница (митигировано: разные режимы UI)

## Альтернативы

### 1. Единый механизм через Cache API
- ✗ Избыточен для Step-by-Step — не нужно сохранять версии в `.cache/` при интерактивной работе
- ✓ Единообразие API

### 2. Перезапуск только текущего шага
- ✗ Пользователь не может вернуться к более раннему шагу (например, сменить модель очистки после генерации лонгрида)
- ✓ Проще реализация
