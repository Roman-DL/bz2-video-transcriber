# ADR-016: Надёжность парсинга JSON из LLM

**Статус:** Принято
**Дата:** 2026-02-20

## Контекст

Claude иногда возвращает невалидный JSON — неэкранированные кавычки в русском тексте, обрезанные структуры, пропущенные запятые. Проблема особенно критична для Summary, который генерируется одним LLM-вызовом со сложной JSON-схемой (7 полей). Сбой парсинга = полностью пустой конспект.

Обнаружено при обработке реального транскрипта: Claude вернул 5084 символа валидного текста, но `json.loads()` упал с `Expecting ',' delimiter` на позиции 3702. Вероятная причина — неэкранированная кавычка внутри русской цитаты.

Longread устойчивее благодаря map-reduce (8 независимых вызовов, сбой одного не ломает остальные). Summary — единственный вызов, падение = пустой результат.

## Решение

Двухуровневая защита:

### 1. json-repair в parse_json_safe() (глобально)

Библиотека `json-repair` пытается исправить типичные ошибки JSON (неэкранированные кавычки, отсутствующие запятые, незакрытые скобки). Интегрирована в `parse_json_safe()` — все генераторы получают выгоду автоматически.

### 2. 1 retry в SummaryGenerator

Если json-repair не помог — повторить LLM вызов один раз. Стоимость retry (~$0.01) несравнима с потерей результата. Tokens_used суммируются для корректной статистики.

## Альтернативы

- **Structured output (Claude tool_use API)** — гарантирует валидный JSON, но требует существенной переработки промптов и логики генерации. Рассмотреть если текущее решение окажется недостаточным.
- **Несколько retry** — избыточно, одного достаточно (ошибка обычно детерминированная или случайная, но не повторяющаяся).

## Последствия

- Добавлена зависимость `json-repair` в requirements.txt
- `parse_json_safe()` теперь пытается repair перед возвратом default
- SummaryGenerator делает 1 retry при пустом результате
- Логирование: `"JSON repaired successfully"` и `"Retry succeeded"` для мониторинга
