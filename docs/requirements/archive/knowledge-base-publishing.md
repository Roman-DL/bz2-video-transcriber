# Публикация материалов в базу знаний

> Осмысленные имена файлов + автоматизация цепочки: архив → GDoc → БЗ

**Статус:** Draft
**Дата:** 2026-02-25
**Проекты:** bz2-video-transcribe, md2gdoc, bz2-bot

---

## 1. Проблема

После обработки записей мероприятий система сохраняет результаты в архив. Далее материалы нужно опубликовать в базу знаний: лонгрид преобразовать в Google Doc, создать материал в БЗ и загрузить чанки. Сейчас этот процесс неудобный и ручной.

**Текущая ситуация:**

Три независимых проекта участвуют в цепочке:

| Проект | Роль | Текущее взаимодействие |
|--------|------|------------------------|
| **bz2-video-transcribe** | Обработка записей → архив | Сохраняет файлы с фиксированными именами |
| **md2gdoc** | Конвертация MD → GDoc | Сканирует архив по polling-правилу, ищет `longread.md` |
| **bz2-bot** | Управление базой знаний | Ручное создание материала + загрузка чанков |

Текущий workflow:

```
1. bz2-video-transcribe обрабатывает запись
   → archive/2026/ПШ/04.07/SV Группа поддержки (Дмитрук)/
       longread.md, summary.md, transcript_chunks.json, ...

2. md2gdoc (polling) находит longread.md → создаёт GDoc
   → GDoc получает название "longread" (из имени файла)

3. Администратор вручную в портале bz2-bot:
   a. Создаёт Material с типом topic_longread
   b. Указывает название, URL на GDoc, секцию
   c. Загружает файл transcript_chunks.json
   d. Проверяет чанки, активирует материал
```

**Почему это проблема:**

- **Фиксированные имена файлов.** Все лонгриды — `longread.md`, все саммари — `summary.md`. В Obsidian (при подключении архива по симлинку) невозможно различить файлы. GDoc получает бессмысленное название "longread". Нет возможности быстро идентифицировать содержимое файла без открытия папки
- **Ручной процесс публикации.** 4 шага в админ-портале для каждого материала. Нужно вручную вводить название, копировать URL GDoc, выбирать и загружать JSON. Метаданные (спикер, мероприятие, дата) уже есть в архиве, но вводятся заново
- **Разрыв между обработкой и публикацией.** Обработка записей и решение о публикации — разные бизнес-решения с разным таймингом. Администратор может обработать записи сразу, а опубликовать позже. Но текущий процесс не поддерживает удобный отложенный импорт из архива

---

## 2. Решение

### Ключевые идеи

1. **Осмысленные имена MD-файлов** — лонгриды, саммари и истории сохраняются с человекочитаемыми именами вместо фиксированных. Технические файлы (JSON, TXT) остаются с фиксированными именами
2. **Импорт из архива в bz2-bot** — новый flow в админ-портале для типа `topic_longread`: указать папку → система находит артефакты, конвертирует лонгрид через md2gdoc API, создаёт материал и индексирует чанки
3. **md2gdoc как сервис** — используется по API (`POST /api/convert`), не как polling-правило для лонгридов из transcriber
4. **Только для лонгридов** — автоматический импорт из архива реализуется только для `topic_longread`. Для `topic_video` остаётся текущий ручной процесс (видео выкладывается на стриминговый сервис, URL вводится вручную)

### Два типа материалов — два flow публикации

| Тип | Источник | URL материала | Flow публикации |
|-----|----------|---------------|-----------------|
| `topic_longread` | Аудиозапись или транскрипт | Ссылка на GDoc (автоматически через md2gdoc) | **Новый:** импорт из архива |
| `topic_video` | Видеозапись | Ссылка на стриминговый сервис (вручную) | **Текущий:** ручное создание в админ-портале |

Оба типа генерируют одинаковый `transcript_chunks.json`. Разница — в источнике URL для материала и в степени автоматизации.

### Разделение ответственности

```
bz2-video-transcribe  →  "Обработай запись"  →  archive folder
md2gdoc               →  "Конвертируй MD"    →  GDoc URL
bz2-bot               →  "Опубликуй в БЗ"    →  Material + embeddings
```

Решение о публикации принимается администратором в портале bz2-bot. Обработка записей не привязана к публикации — можно обработать пакет записей после мероприятия, а опубликовать позже.

### Архитектура

```
                    bz2-video-transcribe
                            │
                    обработка записи
                            │
                            ▼
            ┌─── archive folder ──────────────────────┐
            │                                         │
            │  Мужской клуб (Руцман) — лонгрид.md     │
            │  Мужской клуб (Руцман) — саммари.md     │
            │  transcript_chunks.json                 │
            │  transcript_cleaned.txt                 │
            │  pipeline_results.json                  │
            └──────────────┬──────────────────────────┘
                           │
              Obsidian (ревью, правки)
                           │
                           ▼
              ┌─── bz2-bot админ-портал ────┐
              │                             │
              │  [Импорт из архива]         │
              │  Указать папку → Превью     │
              │  [Создать материал]         │
              │                             │
              └──────┬──────────┬───────────┘
                     │          │
          ┌──────────┘          └──────────┐
          ▼                                ▼
   md2gdoc API                    bz2-bot backend
   POST /api/convert              Create Material
   MD → GDoc URL                  Import Chunks
                                  Index Embeddings
```

## 3. Пользовательский сценарий

### Сценарий A: Обработка записи (bz2-video-transcribe)

```
1. Пользователь загружает видео или MD-файл
2. Pipeline обрабатывает материал
3. Результат сохраняется в архив:
   archive/2026/ПШ/04.07/SV Группа поддержки (Дмитрук)/
     ├── Группа поддержки (Дмитрук) — лонгрид.md
     ├── Группа поддержки (Дмитрук) — саммари.md
     ├── transcript_chunks.json
     ├── transcript_cleaned.txt
     ├── pipeline_results.json
     └── ...
```

### Сценарий B: Ревью в Obsidian (промежуточный)

```
1. Архив подключен к Obsidian по симлинку
2. В сайдбаре видно:
   - "Группа поддержки (Дмитрук) — лонгрид"
   - "Группа поддержки (Дмитрук) — саммари"
   - "Мужской клуб (Руцман) — лонгрид"
3. Администратор открывает, читает, вносит правки при необходимости
4. Когда материал готов — переходит к публикации
```

### Сценарий C: Публикация лонгрида в БЗ (bz2-bot, topic_longread)

```
1. Администратор открывает "Добавить материал"
2. Выбирает тип "Лонгрид темы"
3. Нажимает кнопку [Загрузить из папки] (новая)
4. Выбирает папку из дерева архива (или вводит путь)
5. Система автоматически:
   a. Читает MD-файл лонгрида из папки
   b. Вызывает md2gdoc API → получает GDoc URL → подставляет в поле URL
   c. Читает transcript_chunks.json → загружает как JSON с чанками
   d. Запускает AI Анализ → заполняет Название, Раздел, Описание
6. Администратор видит заполненную форму, проверяет данные
7. Нажимает "Создать" — стандартный процесс создания материала
```

> Кнопка "Загрузить из папки" — это автозаполнение существующей формы, а не замена процесса. Администратор видит те же поля и может их скорректировать перед созданием. Текущий ручной flow (вставить URL + загрузить JSON) остаётся полностью рабочим.

### Сценарий D: Публикация видеоматериала в БЗ (bz2-bot, topic_video)

Текущий ручной процесс остаётся без изменений:

```
1. Администратор выкладывает видео на стриминговый сервис
2. В админ-портале создаёт материал с типом "Видео"
3. Указывает название, URL стримингового сервиса, секцию
4. Загружает файл transcript_chunks.json из архива
5. Проверяет чанки, активирует материал
```

> Автоматизация для `topic_video` не реализуется, так как загрузка видео на стриминговый сервис — внешний процесс, не поддающийся автоматизации через API.

## 4. Изменения по проектам

### 4.1 bz2-video-transcribe — именование файлов

**Что меняем:** MD-файлы сохраняются с осмысленными именами. Технические файлы — без изменений.

**Формат имени:** `{title} ({speaker}) — {тип}.md`

| Было | Станет | Тип контента |
|------|--------|--------------|
| `longread.md` | `Группа поддержки (Дмитрук) — лонгрид.md` | educational |
| `summary.md` | `Группа поддержки (Дмитрук) — саммари.md` | educational |
| `story.md` | `История (Иванов Дмитрий) — история.md` | leadership |

**Без изменений (технические файлы):**

| Файл | Причина |
|------|---------|
| `transcript_chunks.json` | Потребляется bz2-bot по API, не виден в Obsidian |
| `pipeline_results.json` | Потребляется Web UI transcriber |
| `transcript_raw.txt` | Технический артефакт |
| `transcript_cleaned.txt` | Технический артефакт |
| `audio.mp3` | Медиафайл |

**Источник данных для имени:** `VideoMetadata` — уже содержит `title` и `speaker`, распарсенные из имени входного файла на этапе Parse.

**Затрагиваемые файлы:**
- `backend/app/services/saver.py` — формирование имён файлов при сохранении (FileSaver)
- `backend/app/services/pipeline/runner.py` — если имя файла передаётся в результат
- `frontend/` — если UI показывает имена файлов

**Обратная совместимость:** md2gdoc правило для лонгридов (если используется polling) нужно обновить: паттерн `longread.md` → `*— лонгрид.md`. Либо перейти полностью на API-вызов из bz2-bot.

### 4.2 md2gdoc — доработка API и Docker-сети

API endpoint `POST /api/convert` существует, но требует доработки для надёжной работы при вызове из bz2-bot.

**Контракт API:**

Запрос:
```
POST /api/convert
{
  "mdContent": "<содержимое лонгрида>",
  "fileName": "Группа поддержки (Дмитрук) — лонгрид",
  "targetFolderId": "1abc...",
  "frontmatterMode": "strip"
}
```

Успех (201):
```json
{
  "documentId": "1xyz...",
  "documentUrl": "https://docs.google.com/document/d/1xyz.../edit",
  "status": "success",
  "durationMs": 1250
}
```

Ошибка (4xx/5xx):
```json
{
  "status": "error",
  "errorCode": "AUTH_ERROR",
  "errorMessage": "OAuth2 token expired",
  "durationMs": 45
}
```

Коды ошибок:

| HTTP | errorCode | Причина |
|------|-----------|---------|
| 401 | AUTH_ERROR | OAuth2 токен истёк или недействителен |
| 403 | PERMISSION_ERROR | Нет доступа к папке на Google Drive |
| 404 | NOT_FOUND | Папка не найдена на Google Drive |
| 422 | CONVERSION_ERROR | Ошибка конвертации Markdown |
| 422 | (Pydantic) | Невалидный запрос (стандартный FastAPI формат) |
| 429 | RATE_LIMIT | Rate limit Google API |
| 500 | CONFIG_ERROR | Google OAuth2 не настроен |
| 502 | GOOGLE_API_ERROR | Google API недоступен |

**Проблемы, обнаруженные при интеграции:**

1. **502 Bad Gateway на `POST /api/convert`.** Nginx-фронтенд (md2gdoc-frontend) возвращает 502 при проксировании POST-запроса к бэкенду. Healthcheck (`GET /health`) работает — бэкенд жив, но конвертация недоступна. Нужно проверить и исправить nginx-конфигурацию проксирования API-эндпоинтов.

2. **Docker-сеть.** bz2-admin-api и md2gdoc работают в разных Docker-сетях (`bz2-bot_default` и `md2gdoc_default`). Для вызова по Docker DNS (`http://md2gdoc-frontend:80`) контейнеры должны быть в одной сети. Сейчас подключение настроено вручную (`docker network connect`), но при `docker compose up` это сбрасывается.

**Что нужно сделать:**

| # | Задача | Описание |
|---|--------|----------|
| 1 | Исправить nginx проксирование | `POST /api/convert` должен корректно проксироваться к бэкенду. Проверить upstream, таймауты, `proxy_pass` для `/api/` |
| 2 | Общая Docker-сеть | Создать external network `bz2-shared` и подключить оба compose-проекта. Либо использовать `docker-compose` `networks.external` |
| 3 | Обновить `md2gdoc_url` | После настройки сети — указать правильный адрес: `http://md2gdoc-frontend:80` (внутренний порт) или `http://md2gdoc-backend:8000` (напрямую, без nginx) |

**Вариант с общей Docker-сетью (рекомендуемый):**

```yaml
# docker-compose.yml в каждом проекте
networks:
  bz2-shared:
    external: true

# Создать один раз на сервере:
# docker network create bz2-shared
```

> md2gdoc остаётся универсальным конвертером. Не знает ни про bz2-video-transcribe, ни про bz2-bot. Polling-правило можно удалить после реализации — основной flow через API-вызов из bz2-bot.

### 4.3 bz2-bot — импорт лонгридов из архива

**Новый flow в админ-портале:** "Импорт из архива" — создание материала типа `topic_longread` из папки архива. Доступен только при выборе типа материала "Лонгрид". Текущий ручной процесс создания материалов остаётся как fallback и основной путь для `topic_video`.

#### Backend: новые endpoints

**1. Список папок архива (browse)**

```
GET /api/archive/browse?path=2026/ПШ/04.07
```

Возвращает содержимое папки архива для навигации по дереву. Путь — относительно базового пути архива из настроек.

**Ответ:**

```json
{
  "folders": [
    "SV Группа поддержки (Дмитрук)",
    "НП Работа с возражениями (Иванов)"
  ],
  "isLeaf": false
}
```

Когда папка является конечной (содержит файлы материала), `isLeaf: true`.

**2. Загрузка из папки (конвертация + автозаполнение)**

```
POST /api/archive/load-from-folder
Content-Type: application/json

{
  "archivePath": "2026/ПШ/04.07/SV Группа поддержки (Дмитрук)"
}
```

> Путь относительный от базового пути архива. `md2gdocUrl` и `targetFolderId` — из настроек приложения.

**Логика:**

```
1. Сканировать папку:
   - Найти MD-файл лонгрида (паттерн: *— лонгрид.md или *— история.md)
   - Найти transcript_chunks.json
   - Извлечь метаданные из transcript_chunks.json

2. Конвертировать лонгрид → GDoc:
   - Прочитать MD-файл
   - POST md2gdoc /api/convert (mdContent, fileName, targetFolderId)
   - Получить documentUrl

3. Вернуть данные для автозаполнения формы
```

**Ответ:**

```json
{
  "documentUrl": "https://docs.google.com/document/d/1xyz.../edit",
  "chunksData": { ... },
  "metadata": {
    "title": "Группа поддержки",
    "speaker": "Дмитрук",
    "eventType": "ПШ",
    "stream": "SV",
    "date": "2026-04-07"
  }
}
```

- `documentUrl` → подставляется в поле URL формы
- `chunksData` → содержимое transcript_chunks.json, подставляется как загруженный JSON
- `metadata` → вспомогательная информация для AI Анализа

После этого frontend запускает стандартный flow: AI Анализ → заполнение полей → администратор проверяет → "Создать". Создание Material и импорт чанков происходит через существующий endpoint `POST /api/materials/create-with-chunks`.

#### Frontend: UI

Минимальная доработка существующей формы "Добавить материал":

```
┌─────────────────────────────────────────────────┐
│  ✨ Добавление материала                        │
│                                                 │
│  Тип материала                                  │
│  [Лонгрид темы ▼]                               │
│                                                 │
│  JSON с чанками                                 │
│  [📎 Загрузить JSON]  [📁 Загрузить из папки]   │
                        ^^^^^^^^^^^^^^^^^^^^^^^^^^│
│                        НОВАЯ КНОПКА             │
│                                                 │
│  URL                                            │
│  [Ссылка на видео / лонгрид    ] [AI Анализ]    │
│                                                 │
│  ...остальные поля без изменений...             │
│                                                 │
│                         [Отмена] [Создать]      │
└─────────────────────────────────────────────────┘
```

**Кнопка "Загрузить из папки":**
- Видна только при типе `topic_longread` (или `topic_video` — на будущее)
- По нажатию — открывает browse-диалог или dropdown с деревом папок архива
- После выбора папки вызывает backend (`POST /api/archive/load-from-folder`)
- Backend конвертирует лонгрид через md2gdoc, возвращает URL + JSON-данные
- Frontend подставляет:
  - URL → поле "URL"
  - JSON → как если бы пользователь загрузил файл вручную
- Далее стандартный flow: "AI Анализ" → заполнение полей → "Создать"

**Автоматизация "AI Анализ":** После загрузки из папки можно автоматически запустить AI Анализ, чтобы администратору осталось только проверить и нажать "Создать".

## 5. Структура архивной папки (контракт)

bz2-bot должен уметь парсить структуру архива. Контракт на формат:

```
archive/{year}/{category}/{event_type}/{date}/{folder_name}/
  ├── {title} ({speaker}) — лонгрид.md     # или — история.md
  ├── {title} ({speaker}) — саммари.md      # опционально
  ├── transcript_chunks.json                 # обязателен для импорта
  ├── pipeline_results.json                  # не используется bz2-bot
  ├── transcript_raw.txt                     # не используется bz2-bot
  ├── transcript_cleaned.txt                 # не используется bz2-bot
  ├── audio.mp3                              # не используется bz2-bot
  └── {original_video_file}                  # не используется bz2-bot
```

**Как определить тип контента:**
- Файл `*— лонгрид.md` → `material_type: topic_longread`
- Файл `*— история.md` → `material_type: topic_longread` (leadership-вариант)

**Как извлечь метаданные:**
- Из `transcript_chunks.json` → `metadata` (speaker, event_type, date и т.д.)
- Из имени папки → backup-источник метаданных
- Из frontmatter лонгрида → title, speaker, date, tags

## 6. Ограничения

| Параметр                   | Ограничение                                           | Обоснование                               |
| -------------------------- | ----------------------------------------------------- | ----------------------------------------- |
| md2gdoc доступен по сети   | bz2-bot должен иметь HTTP-доступ к md2gdoc            | Оба на TrueNAS, Docker-сеть               |
| Архив на shared storage    | bz2-bot должен иметь доступ к файловой системе архива | SMB/NFS mount или Docker volume           |
| Google OAuth2 в md2gdoc    | Должен быть настроен заранее                          | Без авторизации конвертация невозможна    |
| Одна папка = один материал | Импорт по одной папке за раз                          | Соответствует структуре архива            |
| Активация — отдельный шаг  | Материал создаётся со статусом `pending`              | Администратор проверяет перед публикацией |
