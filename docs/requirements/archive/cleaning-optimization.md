# Оптимизация очистки транскриптов

> Корректные пороги чанкования + адаптация под Haiku для снижения стоимости и поддержки длинных записей

**Статус:** Done
**Дата:** 2026-02-20
**Версии:** v0.65

---

## 1. Проблема

Очистка транскрипта — самый дорогой этап pipeline по токенам. Текущие пороги чанкования профиля `large` рассчитаны в символах и фактически отключают разбиение для Claude.

**Текущая ситуация:**
- Профиль `large`: `chunk_size: 100000`, `small_text_threshold: 150000` символов
- Часовая запись (43K символов транскрипта) потребляет **180K input tokens** — на грани контекста (200K)
- Глоссарий (34KB ≈ 77K токенов) отправляется с каждым запросом — постоянный overhead
- Русский текст токенизируется неэффективно: **~2.3 токена/символ** (vs ~1.3 для английского)

**Почему это проблема:**
- Записи >1 часа **не помещаются** в контекстное окно 200K → API ошибка, pipeline падает
- Полное заполнение контекста ухудшает качество ответа модели
- Очистка на Sonnet стоит ~$0.83/час записи — неоправданно дорого для механической задачи

---

## 2. Решение

Пересчитать пороги чанкования с учётом реальной токенизации русского текста. Инфраструктура чанкования уже реализована — нужно только скорректировать конфигурацию.

### Ключевые идеи

1. **Корректные пороги** — рассчитать `chunk_size` так, чтобы запрос занимал ~70% контекстного окна (overhead + транскрипт)
2. **Haiku как default для очистки** — задача механическая (удаление слов-паразитов + замена по глоссарию), Haiku справляется на уровне Sonnet за 3x дешевле
3. **Семантический глоссарий** — переход от списков вариаций к описаниям терминов, сокращение overhead на ~40K токенов/чанк

### Расчёт порогов

**Состав запроса очистки (после оптимизации глоссария):**

| Компонент | Было | Стало | Токены (было → стало) |
|-----------|-----:|------:|----------------------:|
| System prompt | ~2K | ~3K | ~4.5K → ~7K |
| Глоссарий (термины) | 34K | ~15K | ~77K → ~35K |
| Глоссарий (люди) | — | ~3K | — → ~7K |
| **Overhead** | **36K** | **~21K** | **~82K → ~49K** |

**Целевое заполнение:** 70% от 200K = 140K токенов

```
До:  140K - 82K (overhead) = 58K токенов ≈ 25K символов на чанк
После: 140K - 49K (overhead) = 91K токенов ≈ 40K символов на чанк
```

Оптимизация глоссария увеличивает допустимый размер чанка **~1.6x**: 25K → 40K символов.

**Прогноз по длительности (Haiku, оптимизированный глоссарий + люди):**

| Запись | Символов | Чанков | Input токены | Стоимость |
|--------|--------:|-------:|------------:|----------:|
| 30 мин | ~22K | 1 | ~100K | ~$0.13 |
| 1 час | ~43K | 2 | ~198K | ~$0.26 |
| 1.5 часа | ~65K | 2 | ~198K | ~$0.26 |
| 2 часа | ~86K | 3 | ~296K | ~$0.39 |

**Сравнение с текущим (Sonnet, без чанков):** 1 час = ~$0.83, 1.5 часа = невозможно.

> Haiku + чанки + оптимизированный глоссарий: **~3.2x дешевле** Sonnet, и работает с любой длиной.

## 3. Формат запроса очистки

### Текущий формат

**System** (`config/prompts/cleaning/system.md`):
- Две задачи: удаление слов-паразитов + терминология по глоссарию
- Правила: 85-95% от входа, не сокращать, сохранять метки спикеров

**User** (`config/prompts/cleaning/user.md`):
```
<glossary>{glossary}</glossary>
<input>{transcript}</input>
```

### Проблемы текущего промпта

Тестирование на реальном транскрипте (часовая запись, спикер с множеством оговорок) показало:
- Обе модели (Sonnet и Haiku) дали **почти одинаковый результат** (+0.5% к объёму)
- Терминология исправлена корректно (Herbalife, VP, Супервайзер)
- Базовые паразиты удалены (ну, значит, как бы)
- **НЕ удалены**: технический шум, дубли предложений, бессвязные фрагменты
- При обработке одним куском появляются **артефакты дублирования** — модель "заедает"

### Решение: добавить задачу 3 — "причёсывание"

В system prompt добавляется **ЗАДАЧА 3** с безопасными, чётко формулируемыми правилами:

1. **Технический шум** — удалить проверку микрофона, счёт ("раз, два, три..."), рассадку в зале
2. **Дословные повторы** — если одно и то же предложение/фраза идёт подряд дважды, оставить один раз
3. **Заикания и заедания** — "Да, сейчас, сейчас, сейчас" → "Да, сейчас"
4. **Незавершённые фразы-обрывки** — если фраза оборвана и тут же начата заново, удалить оборванную часть

**НЕ добавлять** (рискованно для Haiku, оставить для longread/story):
- Переформулировку оборванных мыслей
- Объединение разорванных предложений
- Решения о повторах-для-акцента vs оговорках

### Решённые вопросы по формату

- [x] Номер чанка не нужен — каждый чанк обрабатывается независимо, контекст не требуется
- [x] Оптимизация глоссария — включена в scope (см. секцию 4)
- [x] Дополнительные инструкции для чанков — не нужны, формат user prompt не меняется

## 4. Оптимизация глоссария

### Проблема текущего формата

Глоссарий (79 терминов, 988 строк, 34KB) содержит избыточные данные:

**Типичная запись — 10+ вариаций, из которых полезны 2-3:**
```yaml
- canonical: "Организационный объем"
  english: "Organization Volume"
  description: "Суммарный объем всей организации"
  variations:
    - "организационный объем"     # = canonical в нижнем регистре
    - "объем организации"          # модель поймёт по смыслу
    - "орг объем"                  # полезно
    - "оргобъем"                   # полезно
    - "ОВ"                         # полезно (сокращение)
    - "Organization Volume"        # Whisper не выдаст на русской записи
    - "organization volume"        # то же
    - "OV"                         # то же
    - "OrgVolume"                  # то же
```

**Избыточные типы вариаций:**
- Canonical в нижнем регистре — бессмысленно для LLM
- Английские формы (Organization Volume, OV, OrgVolume) — Whisper транскрибирует русскую речь на русском
- Очевидные производные ("объем организации") — LLM поймёт по описанию

**Реальная проблема:** Whisper выдаёт непредсказуемые ошибки, которые невозможно перечислить. В тестовом транскрипте: "Гербалотово", "Гермалайху", "Дербакхе" — ни одно НЕ было в variations, но модель всё равно исправила по смыслу.

### Решение: семантический формат

Акцент на **описании** (чтобы LLM распознавал по смыслу) вместо **перечисления** (pattern matching):

```yaml
- canonical: "Супервайзер"
  description: "Первый квалификационный уровень в Herbalife. Требуется 2500 VP и группа. Сокращение: СВ (англ. Supervisor, SV)"
  whisper_errors: ["скпервайзер", "супер визор"]
  context: "стал Супервайзером за 3 месяца"
```

**Принципы нового формата:**
1. `description` — расширенное описание: что это, когда упоминается, сокращения (русские и английские)
2. `whisper_errors` — только неочевидные ошибки транскрибации, которые модель не угадает сама
3. `context` — пример использования в речи (помогает модели понять контекст)
4. Убрано: `english` (включено в description), `variations` (заменено на whisper_errors)

**Ожидаемый эффект для терминов:** 34KB → ~15KB (~55% сокращение).

### Раздел `people` — имена ключевых лидеров

**Проблема:** Whisper катастрофически искажает имена людей, и ни Sonnet, ни Haiku не исправляют их без подсказки. В тестовом транскрипте основатель компании Mark Hughes записан **5-6 разными** ошибочными вариантами ("Марк Грюз", "Марк Клюз", "Mark Crews", "Марк Юсс", "Марк Юрс", "Марк Джимрон"). Ни один не был исправлен при очистке.

В отличие от терминологии (где модель угадывает "гербалайф" → "Herbalife" по смыслу), имена людей **невозможно угадать** без справочника — "Марк Грюз" не содержит фонетических подсказок о "Hughes".

**Анализ эталонного транскрипта (Руцман Александр, 1 час):**

| Правильное имя | Whisper варианты | Sonnet исправил? | Haiku исправил? |
|---|---|---|---|
| Mark Hughes (основатель) | Марк Грюз, Марк Клюз, Mark Crews, Марк Юсс, Марк Юрс | Нет (5 вариантов) | Нет (4 варианта) |
| Jim Rohn (философ) | "Марк Джимрон" (склейка с Марк Хьюз) | Нет, не распознан | Нет, не распознан |
| Stephan Gratziani (CEO) | Стэфан Грициани, Степан Трицай, Стэпен Грицани | Нет (3 варианта) | Нет (3 варианта) |
| Michael Johnson (экс-CEO) | Майкл Джонсон | Уже верно | Уже верно |
| Alan Lorenz (Founder's Circle) | Анал Роэнс, Ан Лоренс | Нет | Нет |
| John Tartol (Founder's Circle) | Джон Дартон, Джон Тартал | Нет | Нет |
| David Beckham (амбассадор) | Дэвид Бэрхэн | Нет | Нет |
| Charanjit Singh (лидер Индии) | Чаранжи Цин | Нет | Нет |
| Cristiano Ronaldo (амбассадор) | Клещьяна Рональда (оригинал) | Исправлено | Исправлено |

**Особо критичные проблемы:**
- "Анал Роэнс" (Alan Lorenz) — неприемлемое искажение, попадает в финальный текст
- "Марк Джимрон" — Whisper склеил двух людей (Марк Хьюз + Джим Рон) в одно "имя"
- "Военно-водочной силы" (вместо "военно-воздушных") — фонетически похоже, но абсурдно
- Одно имя (Mark Hughes) записано 5 разными способами в одном тексте — нет консистентности

**Решение:** Добавить раздел `people` в глоссарий. Для имён `whisper_errors` критически важнее, чем для терминов — модель не может угадать правильное имя по смыслу.

```yaml

  - canonical: "Марк Хьюз"
    description: "Mark Hughes — основатель Herbalife (1956–2000). Создал компанию в 23 года"
    whisper_errors: ["Марк Грюз", "Марк Клюз", "Mark Crews", "Марк Юсс", "Марк Юрс"]
    context: "Основатель компании Марк Хьюз"

  - canonical: "Стефан Грациани"
    description: "Stephan Gratziani — CEO Herbalife с 2025. Founder's Circle, бывший велогонщик"
    whisper_errors: ["Стэфан Грициани", "Степан Трицай", "Стэпен Грицани", "Стэпен Грицай"]

    whisper_errors: ["Анал Роэнс", "Ан Лоренс", "Алан Лоренс"]
    context: "его спонсор Алан Лоренц, он Фаундер Серкл"

  - canonical: "Джон Тартол"
    description: "John Tartol — Founder's Circle (#8 в компании), 40+ лет в бизнесе"
    whisper_errors: ["Джон Дартон", "Джон Тартал"]

  - canonical: "Сюзан Питерсон"
    description: "Susan Peterson — Founder's Circle. С 1982 года, многолетний дистрибьютор #1 в мире"
    whisper_errors: ["Сьюзан Петерсон", "Сюзен Питерсон"]
    context: "Сюзан Питерсон, номер один в мире"

  - canonical: "Леон Вайсбейн"
    description: "Leon Waisbein — Chairman's Club, дистрибьютор в 30 странах. Специализация — emerging markets"
    whisper_errors: ["Лион Вайсбейн", "Леон Вайсбайн", "Леон Вайзбейн"]

    description: "Igor & Inna Mekibel — Founder's Circle, одни из первых партнёров в России. 20K"
```

### Масштабируемость глоссария

Глоссарий будет расти (новые имена, термины). Оценка масштабируемости семантического формата:

| Метрика | Текущий (variations) | Семантический 79 терм. | Семантический 120 терм. + 30 людей |
|---|---|---|---|
| Записей | 79 | 79 | ~150 |
| Размер файла | 34KB | ~15KB | ~20KB |
| Токены (overhead) | ~77K | ~35K | ~45K |
| Chunk size (при 70%) | ~25K символов | ~43K символов | ~38K символов |

**Почему масштабируется:**
- `description` растёт линейно O(n) с числом терминов
- `whisper_errors` содержит 0-5 записей (только неочевидные), не 10-15 как `variations`
- Убраны избыточные типы: английские формы, lowercase-дубли, очевидные производные
- При 150 записях overhead ~45K токенов — chunk size ~38K символов, всё ещё достаточно для 30-минутной записи в одном чанке

**Точка масштабирования:** При >200 записей (~25KB, ~55K токенов) chunk size падает до ~35K символов. Если глоссарий превысит этот порог, рассмотреть:
- Разделение на тематические подмножества (отправлять только релевантные секции)
- Compact-формат (убрать `context`, сократить `description` до одного предложения)

На горизонте 120-150 записей эти оптимизации **не требуются**.

## 5. Ограничения

| Параметр | Лимит | Обоснование |
|----------|-------|-------------|
| Заполнение контекста | ≤70% | Запас для качества ответа |
| Токенизация русского | ~2.3 tok/char | Измерено на реальных данных |
| Overhead глоссария (после) | ~45K токенов | System prompt + glossary (термины + люди) |
| Контекст Claude | 200K токенов | Все модели Claude (Haiku/Sonnet/Opus) |
| Chunk size (после) | ~38K символов | (140K target - 45K overhead) / 2.3 tok/char |
| Масштаб глоссария | ≤200 записей | Выше — рассмотреть разделение на подмножества |
