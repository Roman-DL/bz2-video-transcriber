# Переход на single-pass генерацию лонгрида и конспекта

> Убрать map-reduce разбиение текста для облачных моделей с большим контекстом. Использовать готовые шаблоны промптов.

**Статус:** Draft
**Дата:** 2026-02-20
**Версии:** v0.67+ (план)

---

## 1. Проблема

### Текущая ситуация: map-reduce для моделей с 8K контекстом

Лонгрид генерируется через map-reduce pipeline, спроектированный под **локальные модели** (gemma2:9b, 8192 токена):

```
43K текст → TextSplitter (6K частей) → OutlineExtractor (7 LLM вызовов)
         → SectionGenerator (4 LLM вызова) → CombineGenerator (1 LLM вызов)
         = ~12 LLM вызовов на один лонгрид
```

Текущие параметры разбиения (`config/models.yaml`):

| Профиль | part_size | overlap | parts_per_section | Целевая модель |
|---------|-----------|---------|-------------------|----------------|
| small   | 6 000     | 1 500   | 4                 | gemma2:9b (8K) |
| medium  | 15 000    | 3 000   | 6                 | qwen2.5 (32K) |
| **large** | **80 000** | **10 000** | **10** | **Claude (200K)** |

**Проблема:** С v0.29 система использует **только Claude** (200K контекст), но pipeline всё ещё работает через map-reduce. Даже с профилем `large` (part_size=80K) при тексте 43K символов — это 1 часть + overhead outline extraction.

### Почему это проблема

1. **Потеря контекста** — каждая секция видит только свои чанки + outline, а не весь текст. Модель не может строить связи между секциями
2. **Лишние LLM вызовы** — outline extraction, параллельные секции, combine = 6-12 вызовов вместо 1
3. **Точки отказа** — баг с `PartOutline.summary < 10` символов (v0.66 hotfix) — пример: шумный контент ломает outline extraction
4. **Стоимость** — каждый вызов = overhead на system prompt + instructions. Один вызов с полным контекстом дешевле суммы частичных
5. **Качество** — секции генерируются независимо, без обратной связи. Повторы и пропуски неизбежны
6. **Неуниверсальность промптов** — текущие промпты ориентированы на образовательный контент ("факты, рекомендации"), плохо работают с неформальными/философскими темами

### Что НЕ нужно менять

- **Очистка (clean)** — уже оптимизирована в v0.65 (Haiku, chunk 40K). Работает корректно
- **Конспект (summary)** — уже single-pass (1 LLM вызов), но промпты нужно обновить
- **Story** — аналогично summary, single-pass
- **Chunk** — детерминистический (H2 парсинг), без LLM

## 2. Решение

### Ключевые идеи

1. **Single-pass генерация** — отправить ВЕСЬ очищенный транскрипт + инструкции + шаблон в один LLM вызов
2. **Готовые шаблоны** — использовать проработанные инструкции из `docs/template-prompts/` как основу для промптов
3. **Убрать TextSplitter + OutlineExtractor из longread pipeline** — они больше не нужны для облачных моделей
4. **Сохранить context_profiles** — для возможного возврата к локальным моделям, но pipeline по умолчанию использует single-pass

### Архитектура: было → стало

**Было (map-reduce):**
```
Текст → Split(6K) → [Outline(7 вызовов)] → Sections(4 вызова) → Combine(1 вызов)
= 12 LLM вызовов, ~3-5 минут, потеря контекста
```

**Стало (single-pass):**
```
Текст + Instructions + Template → 1 LLM вызов → готовый лонгрид
= 1 LLM вызов, ~1-2 минуты, полный контекст
```

### Прототипы промптов (уже готовы!)

| Документ | Описание |
|----------|----------|
| `docs/template-prompts/Educational_Longread_Instructions.md` | Инструкции для лонгрида: принципы, что убирать/сохранять, правила редактирования, работа со слайдами, разметка для RAG |
| `docs/template-prompts/Educational_Longread_Template.md` | Шаблон выходного формата: YAML-метаданные + структура разделов |
| `docs/template-prompts/Educational_Summary_Instructions.md` | Инструкции для конспекта: вариативные блоки по типу темы (методическая/философская/смешанная), правила извлечения |
| `docs/template-prompts/Educational_Summary_Template.md` | Шаблон конспекта: Obsidian-каллауты, вариативные блоки |

Эти шаблоны **значительно подробнее** текущих промптов и учитывают разные типы контента.

## 3. Пользовательский сценарий

Для пользователя **ничего не меняется** — тот же UI, те же этапы:

```
1. Пользователь загружает видео/транскрипт
2. Система очищает текст (без изменений)
3. Пользователь нажимает "Генерация лонгрида"
4. Система отправляет ВЕСЬ текст в один вызов Claude
5. Результат: лонгрид с YAML-метаданными, разделы по H2
6. Дальше — chunk, describe, save (без изменений)
```

**Разница для пользователя:**
- Быстрее (~1-2 мин вместо ~3-5 мин)
- Качественнее (модель видит весь контекст)
- Универсальнее (работает с любым типом контента)

## 4. Scope: что именно меняется

### Лонгрид

| Компонент | Изменение |
|-----------|-----------|
| `config/prompts/longread/` | Новые промпты на базе `docs/template-prompts/Educational_Longread_*` |
| `backend/app/services/longread_generator.py` | Убрать map-reduce, single-pass генерация |
| `backend/app/services/outline_extractor.py` | Больше не используется для лонгрида (сохранить для summary?) |
| `backend/app/services/text_splitter.py` | Больше не используется для лонгрида |
| `config/models.yaml` → `context_profiles.large.longread` | Пересмотреть параметры |

### Конспект

| Компонент | Изменение |
|-----------|-----------|
| `config/prompts/summary/` | Новые промпты на базе `docs/template-prompts/Educational_Summary_*` |
| `backend/app/services/summary_generator.py` | Обновить промпт-сборку, убрать truncation для large-контекста |

### Не меняется

- API endpoints (формат запроса/ответа)
- UI
- Chunk, Save, Clean, Story, Slides
- Pydantic models (Longread, Summary, LongreadSection)

## 5. Ограничения

| Параметр | Лимит | Обоснование |
|----------|-------|-------------|
| Макс. размер транскрипта | ~350K символов (~120K токенов) | 200K контекст - overhead на промпт (~15K) - output (~30K) |
| Fallback на map-reduce | Если текст > лимита | Для очень длинных видео (3+ часов) |
| Формат вывода | Markdown с YAML front-matter | Для совместимости с chunk (H2 парсинг) |

## 6. Открытые вопросы

- [ ] **Output format**: Текущий pipeline возвращает JSON (`LongreadSection[]`). Шаблоны предполагают Markdown. Нужна ли конвертация или меняем формат?
- [ ] **OutlineExtractor**: Удалить совсем или сохранить для summary (large texts)?
- [ ] **TextSplitter**: Удалить или оставить для fallback на map-reduce (texts > 350K)?
- [ ] **Story generator**: Тоже обновить промпты? У него отдельные шаблоны?
- [ ] **Совместимость кэша**: Старые кэшированные результаты longread в формате `LongreadSection[]` — как мигрировать?
- [ ] **Стоимость**: Single-pass с полным контекстом — дороже одного вызова, но дешевле 12? Нужен расчёт
- [ ] **Параллельная генерация**: Сейчас longread и summary генерируются последовательно. Single-pass позволяет запустить параллельно?
