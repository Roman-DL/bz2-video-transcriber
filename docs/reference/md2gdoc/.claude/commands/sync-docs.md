# Аудит и синхронизация документации

Выполни аудит документации проекта. Найди и устрани накопившийся документационный долг.

> **Скоуп:** Весь проект целиком — "всё ли в порядке?"
> **Для проверки текущих изменений** — используй `/finalize`.
> **Для больших проектов:** используй субагентов чтобы не переполнить контекст.
> **Главная задача:** найти реализованные фичи без документации и устранить разрыв.

---

## Стратегия выполнения

### Оценка размера проекта

Перед началом оцени объём работы:
- Сколько файлов кода в проекте?
- Сколько документов в docs/?
- Насколько сложная структура?

### Выбор подхода

| Размер проекта | Подход |
|----------------|--------|
| Маленький (< 20 файлов) | Прямой анализ в текущем контексте |
| Средний (20-100 файлов) | Субагенты для сбора информации |
| Большой (> 100 файлов) | Субагенты обязательно |

---

## Шаг 0: Карта покрытия

> Ключевой шаг. Выявляет **критический документационный долг** — реализованный код без документации.

### Собрать данные из трёх источников

**1. Код** — какие модули реально существуют в `backend/app/` и `frontend/src/`:
- Какие директории/модули есть
- Какие из них содержат существенную логику (не утилиты)

**2. Документация** — что уже задокументировано:
- `docs/architecture/` — какие модули имеют документы
- `.claude/rules/` — какие модули имеют правила
- `docs/requirements/` — какие требования в каком статусе (Done, In Progress, Draft)

**3. Пересечение** — построить матрицу покрытия:

```markdown
| Модуль/фича | Код | architecture/ | rules/ | requirement |
|-------------|-----|---------------|--------|-------------|
| converter | ✅ | ✅ 01-converter.md | ✅ converter.md | Done |
| rules-engine | ✅ | ✅ 02-rules-engine.md | ✅ rules-engine.md | Done |
| google-drive | ✅ | ✅ 03-google-drive.md | ✅ google-drive.md | Done |
```

### Приоритеты долга

1. **Критический** — модуль реализован (код есть) + requirement Done, но нет architecture/ документа
2. **Значительный** — модуль реализован, но нет rules; или architecture/ документ сильно устарел
3. **Умеренный** — устаревший ARCHITECTURE.md (обзор), отсутствующий ADR
4. **Низкий** — косметика (ссылки, статусы, housekeeping)

## Для больших проектов: использование субагентов

### Шаг 1: Запустить субагентов для сбора информации

Используй Task tool для параллельного сбора данных:

**Агент 1: Анализ кода и модулей**
```
Исследуй структуру проекта:
- Какие компоненты/модули есть в backend/app/ и frontend/src/ (перечисли каждый)
- Какие технологии используются (package.json, requirements.txt)
- Какие паттерны применяются в коде
- Для каждого модуля: краткое описание (1 строка), ключевые файлы

Верни краткий отчёт (не более 500 слов).
```

**Агент 2: Анализ документации, rules и requirements**
```
Прочитай и проанализируй:
- CLAUDE.md — что описано, структура проекта
- docs/ARCHITECTURE.md — какие компоненты описаны
- docs/architecture/ — какие модули задокументированы (список файлов)
- docs/decisions/ — какие ADR есть
- .claude/rules/ — какие модули покрыты правилами (файлы и их paths)
- docs/requirements/ — список требований и их статусы (Done/In Progress/Draft)

Верни краткий отчёт (не более 500 слов).
```

### Шаг 2: Построить карту покрытия

На основе отчётов от субагентов построи матрицу из Шага 0 (см. выше).

### Шаг 3: Детальная проверка + отчёт

На основе карты покрытия и отчётов — выполни проверки из секции "Проверка соответствия" ниже, затем создай отчёт.

## Для небольших проектов: прямой анализ

### Шаг 1: Собрать данные

Прочитай и проанализируй:
- `CLAUDE.md` — entry point, структура проекта
- `docs/ARCHITECTURE.md` — архитектура (обзор)
- `docs/architecture/` — модульная архитектура
- `docs/decisions/` — ADR
- `docs/requirements/` — требования и их статусы
- `.claude/rules/` — правила модулей
- Реальную структуру проекта (backend/, frontend/)

### Шаг 2: Построить карту покрытия

Выполни Шаг 0 (см. выше) — матрица: код → architecture/ → rules → requirements.

### Шаг 3: Проверка соответствия

#### ARCHITECTURE.md (обзор)

1. **Компоненты**: Все ли компоненты из кода описаны в документе?
2. **Структура**: Соответствует ли дерево директорий реальности?
3. **Технологии**: Все ли зависимости из package.json/requirements.txt указаны?
4. **Схема**: Актуальна ли архитектурная диаграмма?

#### docs/architecture/ (модульная архитектура)

1. **Покрытие:** все ли реализованные подсистемы имеют документ?
2. **Актуальность:** соответствуют ли документы реальному коду (таблицы, API, потоки)?
3. **Индекс:** актуален ли `docs/architecture/README.md` (статусы, нумерация)?
4. **Кросс-ссылки:** ссылается ли ARCHITECTURE.md на модульные документы?
5. **Связь с requirements:** есть ли реализованные требования, для которых не создан architecture/ документ?

#### ADR (decisions/)

1. Есть ли недокументированные архитектурные решения в коде?
2. Есть ли устаревшие ADR (код изменился, ADR нет)?
3. Правильно ли расставлены статусы?

#### CLAUDE.md

1. Актуальны ли ограничения?
2. Верна ли структура проекта?
3. Актуальны ли ссылки на документы?

#### .claude/rules/

1. **Покрытие:** все ли активные модули имеют rule-файл?
2. **Актуальность:** соответствуют ли правила реальным паттернам в коде?
3. **Согласованность:** нет ли противоречий между rules и docs/ (ARCHITECTURE.md, ADR)?

#### requirements/ и plans/

1. Есть ли "зависшие" документы (статус не Done, но работа не ведётся)?
2. Все ли завершённые перемещены в archive/?

## Формат отчёта

```markdown
## Отчёт аудита документации

**Дата:** {сегодня}
**Метод:** {прямой анализ / с субагентами}

### Карта покрытия

| Модуль/фича | Код | architecture/ | rules/ | requirement |
|-------------|-----|---------------|--------|-------------|
| ... | ... | ... | ... | ... |

### Критический долг
- {модуль}: реализован, requirement Done, но нет architecture/ документа

### Значительный
- {модуль}: нет rules (паттерны не защищены)
- {документ}: сильно устарел

### Умеренный
- {что устарело и как исправить}

### В порядке
- {что актуально}
```

## Устранение долга

### Оценка объёма

После отчёта оцени объём долга и спроси пользователя:

| Объём долга | Признак | Стратегия |
|-------------|---------|-----------|
| **Малый** | 1-2 пропущенных документа, мелкие обновления | Исправить в текущей беседе |
| **Большой** | 3+ отсутствующих architecture/ документа, системные пропуски | Сохранить план → исправлять по сессиям |

### Стратегия: малый долг — исправить сейчас

Для каждой проблемы из отчёта, **начиная с критических:**

**Отсутствующие architecture/ документы:**
1. Изучи реальный код модуля (первичный источник)
2. Прочитай requirement (если есть — что планировали)
3. Предложи структуру документа пользователю
4. После подтверждения — создай `docs/architecture/{NN}-{module}.md`
5. Обнови `docs/architecture/README.md` (индекс)

**Отсутствующие rules:**
1. Изучи паттерны в реализованном коде модуля
2. Предложи правила на основе фактических паттернов
3. После подтверждения — создай `.claude/rules/{module}.md` с `paths:`

**Остальные проблемы:**
1. Покажи что нужно изменить
2. Спроси подтверждение
3. Внеси изменения

### Стратегия: большой долг — план на несколько сессий

Когда долг слишком велик для одной беседы:

1. **Сохрани план устранения** в `docs/plans/sync-docs-remediation.md`
2. **В текущей беседе** — исправь только то, что укладывается в контекст
3. **В последующих беседах** — ссылайся на план
