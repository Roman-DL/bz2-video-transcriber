# Гайдлайны по документированию

Принципы документирования кода для проекта md2gdoc.

---

## Главный принцип: Код — источник истины

- **Docstrings в коде** всегда актуальны (меняются вместе с кодом)
- **Внешняя документация** (.md файлы) описывает архитектуру и интеграцию
- **ИИ читает код напрямую** — детали реализации не нужно дублировать

---

## Что документировать во внешних .md файлах

| Категория | Пример | Почему нужно |
|-----------|--------|--------------|
| Архитектура | Поток конвертации MD→GDoc | Не видно из одного файла |
| Интеграция | Как Rules Engine связан с Converter | Требует понимания системы |
| Решения | Почему polling, а не inotify | Код не объясняет "почему" |
| Конфигурация | Service Account, polling interval | Меняются при деплое |
| Форматы данных | Формат каллаутов в Google Doc | Контракты между компонентами |
| Деплой | Docker-compose, volumes | Не связано с кодом |

---

## Что НЕ документировать во внешних .md файлах

| Категория | Почему НЕ документируем |
|-----------|-------------------------|
| Сигнатуры методов | Устаревают при рефакторинге |
| Константы и переменные | ИИ прочитает из кода |
| Приватные методы (`_method`) | Детали реализации |
| Примеры кода с конкретными значениями | Копипаст устаревает |
| Номера строк | Меняются при любом редактировании |
| Полные листинги кода | Дублирование → рассинхронизация |

---

## Формат docstrings в коде

Используем Google-style docstrings. Обязательны для **публичных методов**.

```python
def method(self, arg1: Type, arg2: Type) -> ReturnType:
    """
    Краткое описание в одну строку.

    Расширенное описание если нужно (опционально).

    Args:
        arg1: Описание первого аргумента
        arg2: Описание второго аргумента

    Returns:
        Описание возвращаемого значения

    Raises:
        ValueError: Когда выбрасывается это исключение
    """
```

**Для классов:**
```python
class ConverterService:
    """
    Конвертация MD↔HTML↔Google Doc.

    Поддерживает прямую (MD→GDoc) и обратную (GDoc→MD) конвертацию
    с сохранением каллаутов, таблиц и форматирования.

    Example:
        converter = ConverterService(settings)
        html = converter.md_to_html(markdown_text)
        doc_id = await converter.upload_as_gdoc(html, folder_id)
    """
```

## Когда обновлять внешнюю документацию

| Изменение в коде | Обновить docs? | Что обновить |
|------------------|----------------|--------------|
| Новый сервис/модуль | Да | docs/architecture/ |
| Рефакторинг метода | Нет | Только docstring в коде |
| Изменение API endpoint | Да | api-reference.md |
| Новая конфигурация | Да | configuration.md, CLAUDE.md |
| Баг-фикс | Нет | — |
| Архитектурное решение | Да | Зафиксировать "почему" (ADR) |

## Чек-лист при добавлении нового модуля

- [ ] Docstrings для класса и публичных методов
- [ ] Тесты в `tests/`
- [ ] Создать `docs/architecture/NN-{module}.md`
- [ ] Обновить `docs/architecture/README.md` (индекс)
- [ ] Обновить `docs/ARCHITECTURE.md` если меняется общая картина
- [ ] Обновить `CLAUDE.md` если новая конфигурация
- [ ] Создать `.claude/rules/{module}.md` с паттернами

## Примеры хорошей и плохой документации

### Хорошо (архитектура и "почему")

```markdown
## Polling вместо inotify

Мониторинг папок выполняется через periodic scan (polling):
- inotify/kqueue ненадёжны на SMB/NFS mount'ах
- Polling работает одинаково на локальных и сетевых FS
- Интервал per-rule (1–60 мин) естественно вписывается
```

### Плохо (дублирование кода)

```markdown
## Метод scan_folder()

` ` `python
async def scan_folder(self, rule: Rule) -> list[FileChange]:
    pattern = rule.file_pattern
    ...
` ` `

Этот код устареет при первом же рефакторинге.
```
