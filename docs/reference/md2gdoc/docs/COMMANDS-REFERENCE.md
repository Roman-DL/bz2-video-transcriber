# Справочник команд

> Кастомные команды Claude Code для автоматизации workflow разработки

---

## Обзор команд

| Поведение | Автоматизировано | Ручная команда |
|-----------|-----------------|----------------|
| Preflight | Rules загружаются по globs; CLAUDE.md директива проверять архитектуру | `/preflight` — полный анализ совместимости |
| Finalize | CLAUDE.md секция "После реализации" | `/finalize` — глубокая проверка + drift detection |
| Sync docs | — | `/sync-docs` — полный аудит (всегда ручной) |
| Refactor CLAUDE.md | — | `/refactor-claude` — вынос правил в rules (при > 250 строк) |

**80% рутинных проверок автоматизированы** через rules и CLAUDE.md. Команды нужны для глубокого анализа сложных случаев.

## /preflight [задача | план | пусто]

### Назначение

Помочь спроектировать фичу с учётом существующей архитектуры.

### Когда вызывать

- Сложные фичи, затрагивающие несколько модулей
- После планирования в Plan Mode — проверить совместимость
- Перед сложным рефакторингом

### Можно пропустить

- Простой баг (1-2 файла) — rules автоматически загружают контекст
- Рутинные задачи без архитектурных последствий

### Варианты вызова

```bash
# По файлу плана
/preflight docs/plans/converter-mvp.md

# По описанию задачи
/preflight Добавить two-way синхронизацию

# По контексту беседы
/preflight
```

### Что делает

1. Изучает ARCHITECTURE.md и `docs/architecture/` — компоненты, связи
2. Проверяет ADR — учитывает принятые решения
3. Проверяет CLAUDE.md — ограничения
4. **Рекомендует как интегрировать фичу** — где код, какие паттерны
5. Выявляет конфликты
6. Предварительный список документов для обновления

## /finalize

### Назначение

Страховочная сетка — проверяет фактические изменения и актуализирует документацию.

### Когда вызывать

- После сложных фич/фаз — для глубокого аудита
- После изменений без планирования, экспериментов
- При подозрении на drift (документация ≠ код)

### Можно пропустить

- Минорные изменения — секция "После реализации" в CLAUDE.md покрывает базовое

### Что делает

1. Анализирует изменения в коде
2. Проверяет ARCHITECTURE.md — нужно ли обновить
3. Проверяет `docs/architecture/` — новый документ или обновление
4. Проверяет необходимость ADR
5. Проверяет CLAUDE.md — новые ограничения
6. **Drift detection** — сравнивает rules с реализацией
7. Рекомендация релиза (`cz bump`)
8. Housekeeping (archive планов, cleanup)

## /sync-docs

### Назначение

Аудит документации и устранение накопившегося документационного долга.

### Когда вызывать

- После завершения фазы
- Раз в неделю при активной разработке
- Если несколько фич реализованы без `/finalize`

### Что делает

1. **Строит карту покрытия** — матрица: код → architecture/ → rules/ → requirements
2. **Определяет приоритеты долга:**
   - Критический — реализованный модуль без architecture/ документа
   - Значительный — модуль без rules, устаревший документ
   - Умеренный — устаревший обзор, отсутствующий ADR
3. Проверяет каждую область
4. Создаёт приоритизированный отчёт
5. Устраняет долг (с подтверждением)

## /frontend-design

### Назначение

Двухэтапный процесс: дизайн в Pencil → реализация в React + Tailwind код.

### Когда вызывать

- Новый экран или страница
- Значительная переработка UI

### Не нужен для

- Мелкие правки (текст, цвет, добавить кнопку)

### Что делает

1. **Этап 1: Дизайн** — создаёт визуальный макет в Pencil (.pen файл)
2. Показывает скриншот, итерации с пользователем
3. **Этап 2: Код** — после утверждения генерирует React + TypeScript + Tailwind

### Связанные документы

## /refactor-claude

### Назначение

Вынос модульных правил из CLAUDE.md в `.claude/rules/` когда файл превышает лимит.

### Когда вызывать

- CLAUDE.md > 250 строк (лимит 300)
- Правила для конкретных модулей дублируются в CLAUDE.md и rules

### Что делает

1. Анализирует CLAUDE.md — размер, контент
2. Классифицирует каждое правило по маршрутизации (CLAUDE.md / rules / docs)
3. Выносит модульные правила в `.claude/rules/` (с globs)
4. Обновляет CLAUDE.md — удаляет вынесенное
5. Проверяет результат (< 300 строк, нет дублирования)

### Что остаётся в CLAUDE.md

- Описание проекта, Quick Start, Workflow
- Навигация по документации
- Общие ограничения (polling, Google API, Pydantic)
- Структура проекта, маршрутизация правил

## /finalize vs /sync-docs

| | `/finalize` | `/sync-docs` |
|-|-------------|-------------|
| **Что проверяет** | Только затронутые модули | Весь проект целиком |
| **Когда** | После фичи/фазы | Периодически, при накоплении долга |
| **Drift detection** | Для изменённых файлов | Для всех rules |
| **Карта покрытия** | — | Код → architecture/ → rules/ → requirements |

**finalize = что изменилось сейчас**, **sync-docs = найти весь накопившийся долг**.
