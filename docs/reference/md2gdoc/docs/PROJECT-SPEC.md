# md2gdoc — Спецификация проекта

> Документ требований для AI-разработки

**Версия:** 1.3
**Дата:** 2026-02-23
**Статус:** Draft

---

## Проблема

Для загрузки образовательных материалов (лонгридов) в базу знаний БЗ 2.0 требуется ссылка на Google документ. Сейчас конвертация из Markdown выполняется вручную: открыть MD в Obsidian → режим просмотра → скопировать → создать Google Doc → вставить → получить ссылку. Каждый документ — 5-10 минут ручной работы.

Предыдущая попытка автоматизации через n8n оказалась нестабильной и требовала платный CloudConvert для конвертации HTML → DOCX.

**Текущая ситуация:**
- Pipeline bz2-video-transcriber генерирует лонгриды в формате `.md`
- Для загрузки в БЗ нужен Google Doc со ссылкой
- Форматирование теряется при ручном копировании (каллауты, таблицы)

**Почему это проблема:**
- Ручной процесс не масштабируется при увеличении количества материалов
- Ошибки при копировании снижают качество документов
- Автоматизация через n8n не оправдала себя (нестабильность, стоимость CloudConvert)

---

## Решение

Самостоятельный веб-сервис с административной панелью, ядро которого — **система правил (rules)**. Каждое правило описывает задачу конвертации или синхронизации между исходной и выходной папками с выбранным режимом работы.

### Ключевые идеи

1. **Система правил** — пользователь создаёт правила, каждое из которых связывает исходную папку (локальную FS) с выходной папкой на Google Drive и определяет режим обработки
2. **Три режима работы:**
   - **Разовая конвертация** — при появлении нового файла конвертирует его один раз; повторные изменения игнорируются
   - **Односторонняя синхронизация** — при каждом изменении исходного файла обновляет Google Doc; изменения в Google Doc игнорируются
   - **Двусторонняя синхронизация** — отслеживает изменения и в исходной, и в выходной папке; конфликты разрешаются по last-modified timestamp
3. **MD → HTML → Google Doc** — конвертируем MD в HTML, загружаем на Google Drive с автоконвертацией в Google Doc (бесплатно, без CloudConvert)
4. **Мониторинг локальных папок** — периодическое сканирование (polling) директорий на появление новых/изменённых `.md` файлов. Polling надёжен на любых FS (локальных и SMB/NFS mount'ах). Режим `once` — рекурсивный поиск в подпапках; `one-way`/`two-way` — только указанная папка (для вложенных создаются отдельные правила)
5. **Административная панель** — веб-интерфейс для управления правилами, просмотра логов, настройки параметров конвертации
6. **Универсальность** — работает с любыми MD файлами, не привязан к конкретному pipeline

### Правило (Rule) — центральная сущность

Каждое правило содержит:

| Параметр | Описание | Примеры значений |
|----------|----------|-----------------|
| Имя | Человекочитаемое название правила | "Лонгриды → БЗ 2.0" |
| Исходная папка | Локальная FS (серверная или сетевой mount) | `/mnt/main/work/bz2/video/archive/`, `/mnt/obsidian-vault/Drafts/` |
| Выходная папка | Google Drive | `gdrive:folder_id` |
| Режим | Один из трёх режимов работы | `once`, `one-way`, `two-way` |
| Интервал опроса | Частота проверки источника | 1–60 мин (default 5) |
| Рекурсивный поиск | Искать файлы в подпапках | `true` (default для `once`), `false` (default для `one-way`/`two-way`) |
| Фильтр файлов | Какие файлы обрабатывать | `*.md`, `longread.md`, `*лонгрид*` |

### Режимы работы (подробно)

#### 1. Разовая конвертация (`once`)

```
Исходная папка ──[новый файл]──▶ Конвертация ──▶ Google Doc
                                      │
                  Повторные изменения  ✕  (игнорируются)
```

**Use case:** Pipeline bz2-video-transcriber генерирует `longread.md` → нужна ссылка на Google Doc. Файл конвертируется один раз, дальнейшие изменения MD не отслеживаются.

#### 2. Односторонняя синхронизация (`one-way`)

```
Исходная папка ──[новый/изменённый файл]──▶ Конвертация ──▶ Google Doc (обновление)
                                                    │
                  Изменения в Google Doc  ✕  (игнорируются)
```

**Use case:** Автор пишет документ в Obsidian на Mac. Vault доступен серверу через SMB/NFS mount. Каждое сохранение обновляет Google Doc. Коллеги видят актуальную версию, но их правки в Google Doc перезаписываются при следующей синхронизации.

#### 3. Двусторонняя синхронизация (`two-way`)

```
Исходная папка ◀──[синхронизация]──▶ Google Doc
     │                                    │
     └───── Изменения отслеживаются ──────┘
```

**Use case:** Документы хранятся в Obsidian vault на Mac (доступен серверу через SMB/NFS mount). Команда редактирует в Google Docs, изменения синхронизируются обратно в MD файлы в vault. Требуется обратная конвертация Google Doc → MD.

### Среда развёртывания

Сервис разворачивается на **TrueNAS** (Docker). Исходные папки — локальная FS сервера:

| Режим | Исходная папка | Как доступна серверу |
|-------|---------------|---------------------|
| `once` | Серверная папка (`/mnt/main/work/bz2/video/archive/`) | Прямой доступ — папка на том же сервере |
| `one-way` | Obsidian vault на Mac | SMB/NFS mount (например `/mnt/obsidian-vault/`) |
| `two-way` | Obsidian vault на Mac | SMB/NFS mount (например `/mnt/obsidian-vault/`) |

> **Важно:** Для режимов `one-way`/`two-way` с Obsidian vault на Mac необходимо настроить сетевую шару (SMB/NFS) и примонтировать её на сервере. Polling (периодическое сканирование) работает одинаково для любых локальных путей — ему неважно, физический это диск или сетевой mount.

#### Конвертер — pipeline обработки

```
MD → GDoc (прямая конвертация):
  MD → [Parse frontmatter] → [Convert MD → HTML] → [Image Resolution*] → Upload as Google Doc

GDoc → MD (обратная конвертация, two-way):
  Export GDoc as HTML → [Detect callouts by table format] → [Restore local image links] → Convert HTML → MD

* Image Resolution — заложена в архитектуре, MVP: skip + warning
```

## Пользователи и сценарии

### Первичный пользователь

Администратор базы знаний БЗ 2.0 — загружает образовательные материалы (лонгриды, саммари) и нуждается в ссылках на Google Docs.

### Вторичный пользователь

Автор контента — пишет документы в Obsidian, команда редактирует их в Google Docs. Нужна синхронизация изменений в обе стороны.

### Основные сценарии

#### Сценарий 1: Первоначальная настройка

1. Открыть admin panel (https://md2gdoc.home)
2. Загрузить JSON-ключ Service Account (или указать путь к файлу на сервере)
3. Расшарить целевую папку на Google Drive на email Service Account
4. Создать правило:
   - Имя: "Лонгриды → БЗ 2.0"
   - Исходная папка: `/mnt/main/work/bz2/video/archive/`
   - Выходная папка: Google Drive "БЗ 2.0 / Лонгриды"
   - Режим: разовая конвертация (`once`)
   - Фильтр: `longread.md`
   - Интервал: 5 мин
4. Активировать правило

**Результат:** Сервис начинает отслеживать папку по заданному правилу

#### Сценарий 2: Разовая конвертация (режим `once`)

1. Pipeline bz2-video-transcriber сохраняет `longread.md` в `/archive/`
2. Правило обнаруживает новый файл (polling)
3. Конвертирует MD → HTML → загружает как Google Doc
4. Записывает в лог: файл, ссылка, время
5. Помечает файл как обработанный — повторные изменения этого файла игнорируются

**Результат:** Google Doc доступен по ссылке без ручных действий. Файл конвертирован один раз.

#### Сценарий 3: Односторонняя синхронизация (режим `one-way`)

1. Настроить SMB/NFS шару Obsidian vault с Mac на сервер (например `/mnt/obsidian-vault/`)
2. Пользователь создаёт правило:
   - Исходная папка: `/mnt/obsidian-vault/Drafts/`
   - Выходная папка: Google Drive "Published/"
   - Режим: `one-way`
   - Фильтр: `*.md`
3. Автор редактирует MD файл в Obsidian на Mac
4. Правило обнаруживает изменение через polling (файл изменился на сетевом mount)
5. Обновляет существующий Google Doc (сохраняет ссылку)
6. Изменения, сделанные в Google Doc коллегами, будут перезаписаны при следующей синхронизации

**Результат:** Google Doc всегда отражает актуальную версию MD файла. Ссылка на документ стабильна.

#### Сценарий 4: Двусторонняя синхронизация (режим `two-way`)

1. Настроить SMB/NFS шару Obsidian vault с Mac на сервер
2. Пользователь создаёт правило:
   - Исходная папка: `/mnt/obsidian-vault/Команда/`
   - Выходная папка: Google Drive "Team Docs/"
   - Режим: `two-way`
   - Фильтр: `*лонгрид*`
3. Автор создаёт MD файл в Obsidian → конвертируется в Google Doc
4. Коллега редактирует Google Doc
5. Правило обнаруживает изменение Google Doc (polling) → конвертирует обратно в MD → записывает в vault на Mac через mount
6. Автор видит обновлённый файл в Obsidian, вносит правки → обновляется Google Doc
7. При одновременных изменениях — побеждает последняя по времени правка

**Результат:** MD и Google Doc синхронизированы в обе стороны. Команда работает в удобном для каждого инструменте.

#### Сценарий 5: Ручная конвертация (без правила)

1. Открыть admin panel
2. Загрузить MD файл (drag & drop) или указать путь
3. Выбрать целевую папку на Google Drive
4. Нажать "Конвертировать"

**Результат:** Ссылка на созданный Google Doc (разовое действие, не создаёт правило)

## Scope

### Включено (MVP — v0.1: конвертация + базовые правила)

- [ ] MD → HTML конвертер с поддержкой: заголовки H1-H6, bold, italic, списки, таблицы, ссылки, код
- [ ] Обработка Obsidian callouts (`> [!type]`) → HTML таблицы
- [ ] Обработка YAML frontmatter (strip / вставить в шапку документа, настраивается)
- [ ] Google Drive API: Service Account + upload HTML as Google Doc
- [ ] API endpoint: `POST /api/convert` (ручная конвертация)
- [ ] CRUD для правил (API + UI): создание, редактирование, удаление, пауза/возобновление
- [ ] Режим `once` — разовая конвертация новых файлов
- [ ] Polling: периодическое сканирование локальных папок (серверных и сетевых mount'ов)
- [ ] Рекурсивный поиск в подпапках для режима `once` (выгрузка в плоскую папку на GDrive)
- [ ] Фильтр файлов: glob-паттерны (`*.md`, `longread.md`, `*лонгрид*`)
- [ ] Настраиваемый интервал опроса (1–60 мин)
- [ ] Дедупликация по hash (не конвертировать один файл дважды)
- [ ] Фоновый worker для конвертаций
- [ ] Базовый UI: список правил, быстрая конвертация, логи
- [ ] Деплой через Docker-compose (Traefik совместимый)

### Включено (v0.2 — односторонняя синхронизация)

- [ ] Режим `one-way` — обновление Google Doc при изменении исходного файла
- [ ] Обновление существующего Google Doc (сохранение ссылки) вместо создания нового
- [ ] Трекинг связей: MD файл ↔ Google Doc ID (таблица `file_mappings`)
- [ ] UI: визуальное отличие режимов в карточке правила
- [ ] UI: индикация направления синхронизации

### Включено (v0.3 — двусторонняя синхронизация + admin panel)

- [ ] Режим `two-way` — отслеживание изменений и в исходной, и в выходной папке
- [ ] Обратная конвертация: Google Doc → MD
- [ ] Разрешение конфликтов по last-modified timestamp
- [ ] UI: полноценная admin panel (настройки, статус Service Account)
- [ ] UI: индикация конфликтов и история синхронизаций

### Исключено (Out of Scope)

| Что отложено | Почему |
|--------------|--------|
| Google Drive как исходная папка | Все текущие сценарии покрываются локальными путями (серверные + SMB/NFS mount). Добавим отдельным требованием при необходимости |
| Реализация Image Resolution (v0.1) | Заложено в архитектуре, MVP использует placeholder. Реализация upload изображений — позже (см. [Приложение: Формат изображений](#формат-изображений-в-конвертации)) |
| Embedded файлы, dataview | Специфика Obsidian, не используется в pipeline |
| Merge-уровневое разрешение конфликтов | Сложность; в v0.3 — last-modified wins. Ручное разрешение — на будущее |
| Batch upload через UI | Правила с мониторингом папок покрывают этот use case |
| Кастомные CSS темы | Нейтральные стили достаточны, можно добавить позже |
| Версионирование документов | Google Docs имеет встроенную историю версий |

## Технический контекст

### Предпочтительный стек

| Компонент | Технология | Почему |
|-----------|------------|--------|
| Backend | Python, FastAPI | Консистентность с bz2-video-transcriber, async support |
| Frontend | React + Vite + Tailwind | Консистентность с bz2-video-transcriber, опыт со стеком, модальные окна и сложные UI |
| БД | SQLite | Файловая, без внешних зависимостей, достаточна для объёма |
| MD → HTML | Python-Markdown или mistune | Нативный Python, расширяемый (callouts, frontmatter) |
| File watcher | Polling (periodic scan) | Надёжен на любых FS (локальных и SMB/NFS mount'ах), watchdog `PollingObserver` или свой scan |
| Google API | google-api-python-client, google-auth | Официальный SDK |
| Deploy | Docker, Traefik | Рядом с bz2-video-transcriber на том же сервере |

### Ограничения

| Параметр | Лимит | Обоснование |
|----------|-------|-------------|
| Размер MD файла | 10 МБ | Ограничение Google Drive API |
| Google Drive API | 300 запросов/мин | Rate limit Google API |
| Polling интервал | 1-60 мин (default 5) | Баланс скорости и ресурсов |
| Одновременных конвертаций | 3 | Не превышать rate limits |
| Длина имени документа | 255 символов | Ограничение Google Docs |

### Интеграции

- **Google Drive API** — Files.create с `mimeType: application/vnd.google-apps.document` для автоконвертации HTML → Google Doc. Для `two-way` — Files.export для обратной конвертации Google Doc → HTML/DOCX
- **Google Service Account** — JSON-ключ на сервере, автономная авторизация без UI. Целевые папки на Google Drive расшариваются на Service Account
- **bz2-video-transcriber** — как источник MD файлов для режима `once` (серверная папка `/archive/`)
- **SMB/NFS** — сетевой доступ к Obsidian vault на Mac для режимов `one-way`/`two-way` (настраивается вне приложения)

## Глоссарий

| Термин          | Определение                                                                                                                           |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| Rule (правило)  | Центральная сущность системы — задача конвертации/синхронизации между исходной папкой (локальная FS) и выходной папкой (Google Drive) |
| Режим `once`    | Разовая конвертация — новый файл конвертируется один раз, дальнейшие изменения игнорируются                                           |
| Режим `one-way` | Односторонняя синхронизация — каждое изменение исходного файла обновляет Google Doc                                                   |
| Режим `two-way` | Двусторонняя синхронизация — изменения отслеживаются в обе стороны (MD ↔ Google Doc)                                                  |
| Longread        | Структурированный образовательный материал 5-10 страниц, генерируется pipeline bz2-video-transcriber                                  |
| Callout         | Блок-цитата Obsidian формата `> [!type] Title`, используется для выделения важной информации                                          |
| Frontmatter     | YAML-метаданные в начале MD файла (title, speaker, date, tags)                                                                        |
| File mapping    | Связь между исходным MD файлом и созданным Google Doc (для режимов `one-way` и `two-way`)                                             |
| Network mount   | SMB/NFS шара, через которую Obsidian vault на Mac доступен серверу как локальная папка                                                |
| БЗ 2.0          | База знаний — внешняя система, куда загружаются материалы по ссылке на Google Doc                                                     |

## Открытые вопросы

- [x] **Service Account vs OAuth2** — ~~для серверного приложения Service Account проще, но ограничен одним аккаунтом. OAuth2 гибче, но требует UI для авторизации~~ **Решено: Service Account.** Серверное приложение, работает автономно 24/7. Целевая папка на Google Drive расшаривается на Service Account (разовая настройка). Не нужен OAuth flow в UI, токен не протухает
- [x] **Обновление vs создание нового** — ~~при изменении MD файла: обновлять тот же Google Doc (сохраняет ссылку) или создавать новую версию?~~ **Решено:** зависит от режима. `once` — всегда новый. `one-way` и `two-way` — обновлять существующий (через `file_mappings`), сохраняя ссылку
- [x] **Поддержка изображений** — ~~MD файлы из pipeline не содержат изображений, но универсальный инструмент может столкнуться с ними. Отложить или заложить сразу?~~ **Решено: заложить в архитектуру, реализовать позже.** Изображения — локальные пути в Obsidian (`![alt](./images/photo.png)`, `![[image.png]]`). MD → GDoc: фаза Image Resolution (upload на GDrive → подстановка URL). MVP: placeholder + warning в логе. GDoc → MD (обратная): изображения в GDoc не добавляются/не меняются, могут быть удалены — оставить существующие локальные ссылки, удалённые убрать из MD (файлы на диске не трогать). См. [Приложение: Формат изображений](#формат-изображений-в-конвертации)
- [x] **Стек frontend** — ~~React (как в transcriber) или что-то более лёгкое (Svelte, plain HTML) для простого admin panel?~~ **Решено: React + Vite + Tailwind.** Консистентность с bz2-video-transcriber, опыт работы со стеком, поддержка модальных окон и сложных UI компонентов
- [x] **Обратная конвертация Google Doc → MD** — ~~для режима `two-way` нужна конвертация в обратном направлении~~ **Решено: экспорт в HTML → парсинг → MD.** Каллауты распознаются по детерминированному формату в Google Doc: таблица 2×1, первая строка с затенённым фоном + иконка (см. [Приложение: Формат каллаутов](#формат-каллаутов-в-google-doc)). Изображения при обратной конвертации: существующие локальные ссылки сохраняются, удалённые из GDoc — убираются из MD
- [x] **Стратегия разрешения конфликтов (two-way)** — ~~Нужен ли UI для ручного разрешения? Нужно ли хранить историю конфликтов?~~ **Решено: last-modified wins, без дополнительного UI.** Достаточно для текущих сценариев. Факт конфликта логируется в историю операций
- [x] **Рекурсивный мониторинг** — ~~отслеживать ли подпапки исходной директории?~~ **Решено: рекурсивный для `once`, нет для `one-way`/`two-way`.** Режим `once`: рекурсивный поиск в подпапках, выгрузка в плоскую папку на GDrive (без вложенности). Режимы `one-way`/`two-way`: без рекурсии — для разных папок создаются отдельные правила
- [x] **Watchdog на сетевых mount'ах** — ~~watchdog (inotify/kqueue) может не получать события от SMB/NFS mount'ов~~ **Решено: polling (периодическое сканирование).** `inotify`/`kqueue` ненадёжны на сетевых FS. Polling работает одинаково на локальных и сетевых mount'ах. Интервал опроса per-rule (1–60 мин) естественно вписывается. Реализация: `watchdog.PollingObserver` или простой periodic scan по хешам из БД

### Детальное описание API и данных

Базовое описание API endpoints и SQL схемы — см. [REQUIREMENT-md2gdoc-service.md](../Obsidian%20→%20Google%20Docs/REQUIREMENT-md2gdoc-service.md) (секции 5-6).

> **Внимание:** REQUIREMENT-md2gdoc-service.md содержит API в терминологии `watched_folders`. При реализации заменить на `rules` с учётом изменений ниже.

#### Ключевые изменения API (rules vs watched folders)

```
# Было (watched folders):
GET/POST/PATCH/DELETE  /api/folders
POST  /api/folders/:id/pause
POST  /api/folders/:id/resume

# Стало (rules):
GET/POST/PATCH/DELETE  /api/rules
POST  /api/rules/:id/pause
POST  /api/rules/:id/resume
POST  /api/rules/:id/trigger    # ← ручной запуск правила
```

```typescript
// Rule — заменяет WatchedFolder
interface Rule {
  id: string;
  name: string;                    // человекочитаемое название
  sourcePath: string;              // локальный путь на сервере (или сетевой mount)
  targetFolderId: string;          // ID целевой папки на Google Drive
  mode: "once" | "one-way" | "two-way";
  filePattern: string;             // glob-паттерн
  recursive: boolean;              // поиск в подпапках (default: true для once, false для остальных)
  pollingIntervalMin: number;      // per-rule, 1–60
  status: "active" | "paused";
  lastChecked: string | null;
  filesProcessed: number;          // счётчик обработанных файлов
  createdAt: string;
}
```

```sql
-- Таблица rules (заменяет watched_folders)
CREATE TABLE rules (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  source_path TEXT NOT NULL,       -- локальный путь на сервере (или сетевой mount)
  target_folder_id TEXT NOT NULL,
  mode TEXT NOT NULL DEFAULT 'once',  -- 'once' | 'one-way' | 'two-way'
  file_pattern TEXT DEFAULT '*.md',
  recursive BOOLEAN DEFAULT TRUE,   -- поиск в подпапках (default TRUE для once, FALSE для остальных)
  polling_interval_min INTEGER DEFAULT 5,
  status TEXT DEFAULT 'active',
  last_checked TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Связи файлов (для one-way и two-way)
CREATE TABLE file_mappings (
  id TEXT PRIMARY KEY,
  rule_id TEXT NOT NULL REFERENCES rules(id),
  source_path TEXT NOT NULL,
  source_hash TEXT NOT NULL,         -- для определения изменений
  document_id TEXT NOT NULL,         -- Google Doc ID
  document_url TEXT NOT NULL,
  source_modified_at TIMESTAMP,
  target_modified_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(rule_id, source_path)
);
```

### Формат каллаутов в Google Doc

Каллауты конвертируются в детерминированный формат, позволяющий обратный парсинг (GDoc → MD).

**Obsidian (исходный формат):**
```markdown
> [!warning] Важное предупреждение
> Текст каллаута с **форматированием** и [ссылками](url).
```

**Google Doc (HTML таблица):**
```html
<table border="1" cellpadding="8" style="border-collapse: collapse; width: 100%;">
  <tr style="background-color: #FFF3CD;">
    <td>⚠️ <b>Важное предупреждение</b></td>
  </tr>
  <tr>
    <td>Текст каллаута с <b>форматированием</b> и <a href="url">ссылками</a>.</td>
  </tr>
</table>
```

**Правила распознавания при обратной конвертации (GDoc → MD):**
1. Таблица **2 строки, 1 колонка**
2. Первая строка имеет **затенённый фон** (background-color)
3. Первая строка начинается с **иконки** (emoji) перед текстом
4. Если все три условия выполнены — это каллаут, не обычная таблица

**Маппинг иконок → типов каллаутов:**

| Иконка | Тип каллаута | Цвет фона |
|--------|-------------|-----------|
| ℹ️ | `info`, `note` | `#D1ECF1` |
| 💡 | `tip`, `hint` | `#D4EDDA` |
| ⚠️ | `warning`, `caution` | `#FFF3CD` |
| ❌ | `danger`, `error` | `#F8D7DA` |
| 📝 | `example` | `#E2E3E5` |
| ❓ | `question`, `faq` | `#CCE5FF` |
| 📌 | `abstract`, `summary` | `#E2E3E5` |
| 💬 | `quote`, `cite` | `#E2E3E5` |

> Полный список типов и алиасов — в `topaz_nord_constants.js` (референсный проект). Цвета — нейтральные (не Topaz-Nord).

### Формат изображений в конвертации

#### MD → GDoc (прямая)

**Типы ссылок на изображения в MD:**

| Формат | Пример | Обработка |
|--------|--------|-----------|
| Внешний URL | `![alt](https://example.com/img.png)` | Оставить как `<img src="...">` — GDoc подтянет |
| Локальный путь | `![alt](./images/photo.png)` | Image Resolution: upload на GDrive → подставить URL |
| Obsidian wikilink | `![[photo.png]]` | Resolve путь → upload на GDrive → подставить URL |

**MVP (v0.1):** Image Resolution пропускается. Локальные изображения заменяются на placeholder `[Изображение: photo.png]` + warning в логе. Внешние URL — работают как есть.

**Будущая реализация:** `ImageResolver` загружает локальные файлы на Google Drive (в подпапку `_images/` рядом с документом), получает publicUrl, подставляет в HTML перед upload.

#### GDoc → MD (обратная, two-way)

Изображения в Google Doc **не добавляются и не меняются** пользователями. Могут быть **удалены**.

**Стратегия:**
1. При обратной конвертации — сравнить изображения в GDoc с известными из `file_mappings`
2. Изображения, которые остались — **сохранить оригинальные локальные ссылки** в MD (не трогать)
3. Изображения, которые удалены из GDoc — **убрать ссылку из MD** (сам файл на диске не удалять — может использоваться в других документах)
